# Deployment (v0.9)

Two ways to run this: docker-compose (the full architecture — web, worker, beat, Postgres, Redis,
Prometheus, Grafana) for local dev or any VM that can run Docker, or Render (a free hosted demo,
defined as code in `render.yaml`) for a public URL.

**These two are not the same topology**, and that's deliberate — see the note in Option A below
before you go looking for a `worker` service on Render and don't find one.

## Option A — Render (free tier)

`render.yaml` is a [Render Blueprint](https://render.com/docs/blueprint-spec): infrastructure as
code, currently a single Web Service + managed Postgres.

**Why no worker/beat on Render:** Render's free tier only offers Web Services, Static Sites, a
free Postgres, and a free Redis — **Background Worker is a paid-plan-only service type.** Rather
than pay for that just to run a demo, `render.yaml` sets `CELERY_TASK_ALWAYS_EAGER=1` — the exact
same zero-setup mode this project already uses for local dev and the entire pytest suite (144+
tests exercise this code path), not a workaround bolted on for deployment. `.delay()` runs the
task synchronously inside the request instead of publishing to a queue. Concretely, on the free
tier:

- ✅ Everything triggered by a click still works: create/run a pipeline, cleansing, validation,
  anomaly detection, lineage, scorecards, the dashboard.
- ❌ Cron-scheduled pipelines don't fire — `django-celery-beat` needs an actual running worker
  process to tick, which the free tier doesn't offer. That's a paid-tier (`Background Worker`,
  ~$7/mo/service) upgrade, not something this free demo exercises.

### Steps

1. Push this repo to GitHub (Render deploys from a Git remote, not a local checkout).
2. In the Render dashboard: **New +** → **Blueprint** → connect the repo. Render reads
   `render.yaml` and proposes two resources: `dataflow-studio-web`, `dataflow-studio-db`.
3. Before clicking **Apply**, Render prompts for the `sync: false` values it can't template:
   - `FERNET_KEY` — generate with
     `python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`.
     **Do not reuse the local dev fallback** (`config/settings/base.py` derives one from
     `SECRET_KEY` when unset — fine for zero-setup local dev, not for a real deployment where a
     leaked `SECRET_KEY` would also leak this).
   - `CORS_ALLOWED_ORIGINS` — leave blank for now; comes back in step 5.
   - `DJANGO_SECRET_KEY` is auto-generated by Render (`generateValue: true`) — no action needed.
4. Apply. Render builds the Docker image and runs `python manage.py migrate` automatically
   (`Dockerfile`'s `CMD`). Nothing else to seed — register a user from the frontend (or
   `POST /api/v1/auth/register/`) once it's live.
5. **Frontend:** deploy `frontend/` separately as a Render **Static Site** (build command
   `npm ci && npm run build`, publish directory `dist`), with `VITE_API_BASE_URL` set to the web
   service's URL. Then go back to the web service's env vars and set `CORS_ALLOWED_ORIGINS` to the
   static site's URL; redeploy.
6. (Optional) Add `RENDER_DEPLOY_HOOK_URL` as a GitHub repo secret (Settings → Secrets and
   variables → Actions) — the URL is under the web service's Settings → Deploy Hook in Render.
   With it set, `.github/workflows/ci.yml`'s `deploy` job auto-redeploys on every push to `main`.

**Free-tier caveats** (worth knowing, not blockers): the web service spins down after 15 minutes
idle (first request after that takes ~30-60s to cold-start); the free Postgres instance expires
after 30 days unless upgraded; and anything saved via the "upload a CSV" data source feature
(`POST /api/v1/datasources/upload/`) is written to the container's local disk, which Render wipes
on every redeploy or restart — an uploaded file only lasts for the life of the current container,
not indefinitely. Fine for a portfolio demo, not for anything real.

### Testing the live demo

Once both services are up, walk through this to prove every v0.9 feature works on the deployed
instance, not just locally:

1. **Register + create a workspace** through the frontend (or `POST /api/v1/auth/register/` then
   `POST /api/v1/workspaces/`).
2. **Cleansing:** create a `FILE` data source pointing at `sample_data/customers_dirty.csv` (baked
   into the image — no upload needed). Build a pipeline with a cleansing step: trim `first_name`
   and `email`, lowercase `email`, fill `country` with `UNKNOWN`. Run it — the row with a mixed-case,
   padded email and a missing country should load clean. Check the run's metrics for a `clean` key
   with non-zero `cells_trimmed`/`cells_lowercased`/`cells_filled`.
3. **Anomaly detection:** create a second data source pointing at
   `sample_data/customers_ltv_baseline.csv`, build a pipeline targeting `customers` (same target as
   any other customers pipeline — anomaly baselines are per-dataset, not per-pipeline), run it.
   Then create a third data source pointing at `sample_data/customers_ltv_anomaly.csv` (one row has
   `lifetime_value=50000` against a baseline mean of ~100), build another pipeline also targeting
   `customers`, and run it. Check the Dashboard's "Data anomalies" panel (or
   `GET /api/v1/metadata/anomalies/`) — the $50,000 row should be flagged with a z-score in the
   thousands.
4. **S3 connector:** this one needs a real S3-compatible endpoint the deployed instance can reach
   over the internet — MinIO in `docker-compose.yml` is `localhost`-only, so it can't be reached
   from Render. Point a data source at a real AWS S3 bucket (`bucket`, `key`, `aws_access_key_id`,
   `aws_secret_access_key`, no `endpoint_url`) if you have one, or skip this one on the live demo
   and cite the local docker-compose + MinIO verification instead (see `docs/04-modules.md`'s
   `datasources` section) — it's the same code path either way.

## Option B — docker-compose (local or any Docker host)

```bash
cp .env.example .env
# fill in FERNET_KEY for anything beyond a quick local look (see .env.example)
docker compose up -d --build
```

Brings up `web` (:8000), `worker`, `beat`, `db` (Postgres :5432), `redis` (:6379), `minio` (:9000,
console :9091, free S3-compatible storage — see the `datasources` section of
`docs/04-modules.md`), `prometheus` (:9090), and `grafana` (:3000, default login admin/admin — set
via `GF_SECURITY_ADMIN_PASSWORD` in `docker-compose.yml`). This is the *real* architecture — a
genuine async worker, not eager mode — and is what `loadtest/README.md` and
`scripts/chaos_test.py` target. If you want to demo cron-scheduled pipelines or the chaos-recovery
behavior, this is the option that actually exercises them; Option A's free-tier simplification
does not.

## CI/CD

`.github/workflows/ci.yml` runs on every push to `main` and every PR: a `backend` job (ruff, black
--check, pytest with coverage), a `frontend` job (`npm run lint`, `npm run build`), and a `deploy`
job that only fires on `main` and only if the `RENDER_DEPLOY_HOOK_URL` repo secret is set. Without
that secret the job logs a message and exits 0 (green) rather than failing a fork/PR build that has
no deploy access.
