# Deployment (v0.9)

DataFlow Studio ships as four processes sharing one Docker image (`Dockerfile`): the web API
(gunicorn), a Celery worker, Celery beat, plus managed Postgres and Redis. Two ways to run that:
docker-compose (local, or any VM that can run Docker) or Render (a hosted free tier, defined as
code in `render.yaml`).

## Option A — Render (free tier)

`render.yaml` is a [Render Blueprint](https://render.com/docs/blueprint-spec): infrastructure as
code for the whole stack (web, worker, beat, Postgres, Redis).

1. Push this repo to GitHub (Render deploys from a Git remote, not a local checkout).
2. In the Render dashboard: **New +** → **Blueprint** → connect the repo. Render reads
   `render.yaml` and proposes the five resources it defines (`dataflow-studio-web`,
   `dataflow-studio-worker`, `dataflow-studio-beat`, `dataflow-studio-redis`,
   `dataflow-studio-db`).
3. Before clicking **Apply**, Render will prompt for the `sync: false` values — the ones that
   can't come from a template because they're secrets or environment-specific:
   - `FERNET_KEY` — generate with
     `python -c "from cryptography.fernet import Fernet; print(Fernet.generate_key().decode())"`.
     **Do not reuse the local dev fallback** (`config/settings/base.py` derives one from
     `SECRET_KEY` when unset — fine for zero-setup local dev, not for a real deployment where a
     leaked `SECRET_KEY` would also leak this).
   - `CORS_ALLOWED_ORIGINS` — the deployed frontend's origin, e.g.
     `https://dataflow-studio-frontend.onrender.com`. Leave it blank until the frontend is also
     deployed (see below), then update and redeploy.
   - `DJANGO_SECRET_KEY` is auto-generated by Render (`generateValue: true`) — no action needed.
4. Apply. Render builds the Docker image once and reuses it for `web`, `worker`, and `beat`
   (`dockerCommand` overrides the entrypoint per service — see `render.yaml`).
5. First deploy runs `python manage.py migrate` automatically (`Dockerfile`'s `CMD`). Nothing else
   to seed — register a user from the frontend (or `POST /api/v1/auth/register/`) once it's live.
6. **Frontend:** Render's Blueprint spec doesn't cover static-site services in a way this repo's
   `render.yaml` defines yet — deploy `frontend/` separately as a Render **Static Site** (build
   command `npm ci && npm run build`, publish directory `dist`), with `VITE_API_BASE_URL` set to
   the web service's URL. Then circle back to step 3 and set `CORS_ALLOWED_ORIGINS` to the static
   site's URL.

**Free-tier caveats** (worth knowing, not blockers): Render's free web services spin down after
15 minutes idle (first request after that takes ~30-60s to cold-start); free Postgres/Redis
instances expire after 30/90 days respectively unless upgraded. Fine for a portfolio demo, not for
anything real.

## Option B — docker-compose (local or any Docker host)

```bash
cp .env.example .env
# fill in FERNET_KEY for anything beyond a quick local look (see .env.example)
docker compose up -d --build
```

Brings up `web` (:8000), `worker`, `beat`, `db` (Postgres :5432), `redis` (:6379), `prometheus`
(:9090), and `grafana` (:3000, default login admin/admin — set via `GF_SECURITY_ADMIN_PASSWORD`
in `docker-compose.yml`). This is also what
`loadtest/README.md` and `scripts/chaos_test.py` target — see those for how to load- and
chaos-test the exact stack that ships.

## CI/CD

`.github/workflows/ci.yml` runs on every push to `main` and every PR: a `backend` job (ruff, black
--check, pytest with coverage), a `frontend` job (`npm run lint`, `npm run build`), and a `deploy`
job that only fires on `main` and only if the `RENDER_DEPLOY_HOOK_URL` repo secret is set — get
that URL from the `dataflow-studio-web` service's Settings → Deploy Hook in the Render dashboard,
add it under repo Settings → Secrets and variables → Actions. Without that secret the job logs a
message and exits 0 (green) rather than failing a fork/PR build that has no deploy access.
