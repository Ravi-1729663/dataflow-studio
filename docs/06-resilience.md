# Resilience testing (v0.9)

Two kinds of "does this actually hold up" evidence, beyond the pytest suite: a load test (does it
stay fast under concurrent users?) and a chaos test (does it survive a worker dying mid-task?).

## Load test

`loadtest/locustfile.py` (Locust) drives the same calls the React SPA makes — see
[`loadtest/README.md`](../loadtest/README.md) for how to run it and the full methodology. It
targets the real docker-compose stack (Postgres + Redis + a real Celery worker), not
`manage.py runserver`, since the dev server is single-threaded and would just measure its own
concurrency ceiling.

**Latest result:** 25 concurrent users, 90 seconds, **1203 requests, 0 failures**, p50 19ms / p95
88ms / p99 260ms (dominated by register/login's deliberately-slow password hashing — every other
endpoint stays under 50ms at the median). Full breakdown and the raw CSVs are in
[`loadtest/reports/`](../loadtest/reports/).

## Chaos test

`scripts/chaos_test.py` answers the acceptance criterion directly: **does a run recover cleanly if
the worker processing it dies mid-flight?** It:

1. Builds the `worker` image with a large (20,000-row) synthetic CSV baked in, so a run takes long
   enough to interrupt mid-load.
2. Triggers a real pipeline run against the docker-compose `worker` container.
3. Once the run is `RUNNING`, hard-kills the worker container (`docker kill` — no graceful
   shutdown, simulating a crash/OOM/spot-instance reclaim).
4. Confirms the run is stuck (nothing is consuming it) — this is expected, not a bug.
5. Brings up a replacement worker container (simulating a redeploy or autoscaler).
6. Polls until the run reaches a terminal status, and asserts it's `SUCCEEDED` with every row
   loaded.

**Latest result: PASS.** Full report (regenerated by the script on every run):
[`docs/reports/chaos-test-report.md`](reports/chaos-test-report.md).

### Why it recovers

Three settings in `config/settings/base.py` make this possible:

- `CELERY_TASK_ACKS_LATE = True` — the broker only removes a task's message once a worker
  actually finishes it, not the moment it's handed out. A worker that dies mid-task never acks, so
  the message isn't lost.
- `CELERY_TASK_REJECT_ON_WORKER_LOST = True` — makes that requeue immediate rather than waiting
  purely on the visibility timeout.
- `CELERY_BROKER_TRANSPORT_OPTIONS = {"visibility_timeout": ...}` — bounds how long Redis waits
  before assuming the original consumer died and redelivering the message. Production should set
  this comfortably above the longest real task duration (`CELERY_VISIBILITY_TIMEOUT` env var,
  default 3600s); the chaos test overrides it to 10s so recovery is observable in seconds.

The redelivered task re-invokes `pipelines.tasks.run_pipeline_task` with the *same* `run_id` and
re-runs the whole extract → validate → transform → load from scratch. That's safe only because
every load in this project is an idempotent upsert (`apps/warehouse/services.py`) — re-running an
interrupted load never double-counts or duplicates rows.

**Known caveat:** idempotency covers the *data*. A run that had already sent a notification or
incremented a Prometheus counter before being killed could double those side effects on the
redelivered attempt. Not exercised by this test (the kill happens before either fires in this
scenario) but worth knowing if you extend it.

**Operational nuance found while building this:** under `--pool=solo`, Kombu's Redis transport
only re-scans for expired unacked messages once, at connection time — it doesn't poll on a
recurring timer while otherwise idle. A replacement worker that reconnects *before* the visibility
window has fully elapsed won't pick the stuck message up until something else triggers another
scan. `scripts/chaos_test.py` deliberately waits out the full visibility timeout before starting
the replacement worker for exactly this reason.
